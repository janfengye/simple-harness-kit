# v0.11.0 draft: Phase 2 Quality Engineering Gate

This is the draft release note for the Phase 2 PR. It should be used after the PR is merged and tagged; do not backfill these capabilities into v0.10.0.

## Short version

SHK now treats AI delivery as a quality engineering problem, not just a checklist problem.

The agent must be able to explain:

1. what the iteration is supposed to change;
2. how that change will be tested;
3. whether the tests actually cover the changed risk;
4. whether broken critical behavior would be caught;
5. whether the work can be handed off now.

If the answer is “tests passed, but they did not prove the change,” SHK reports `NOT_SUFFICIENT` and blocks delivery.

## What users will notice

When a user asks Claude Code or Codex to implement a feature, SHK now expects the agent to work like this:

```text
First I will write the iteration spec and identify the risky paths.
Then I will check the project’s existing tests.
If the target app has no E2E for this path, I will generate the smallest useful one.
After it passes, I will check whether it is sufficient: positive path, negative path, assertions, fresh evidence, and mutation/fault proof.
Only then will I say the work is ready.
```

The user should not need to remember `node scripts/shk.js ...` commands. Those commands are backend probes for the AI workflow.

## Major changes

### 1. Spec-driven iteration gate

Medium/high/release tasks depend on `.harness/iteration-spec.json`.

The spec needs to map:

- requirements;
- proposed approach;
- risks;
- traffic paths;
- test plan;
- acceptance evidence.

If the spec is missing, empty, or not connected to the task, the work is not ready to start or deliver.

### 2. Target-app E2E bootstrap

When SHK is installed into a new application project, the AI should inspect the project and decide what kind of E2E makes sense:

- Web or fullstack app: usually Playwright, unless Cypress already exists.
- API service: API E2E, not a fake browser test.
- Unknown project: ask one concrete startup question instead of guessing broadly.

The goal is not to generate lots of tests. The goal is to generate the smallest test that proves the risky path.

### 3. E2E sufficiency

E2E PASS no longer means “safe to ship.”

SHK checks whether the E2E has proof strength:

- real assertions;
- positive path;
- negative or blocking path;
- structured evidence from the current run;
- mapping back to the iteration spec;
- not smoke-only;
- mutation/fault evidence that broken behavior is caught.

Fake E2E, `echo ok`, opening only the home page, or checking only `/health` cannot make medium/high/release work READY.

### 4. Test effectiveness gate

`shk test effectiveness` looks at the relationship between the spec and the tests.

It asks the engineering question users actually care about:

```text
Do these tests prove this change, or did they merely run?
```

### 5. Bounded repair loop

When tests fail or evidence is insufficient, the AI enters a controlled repair loop:

- at most 3 rounds;
- one failure point per round;
- rerun the smallest relevant test;
- stop and explain if no progress is made;
- no auto-push, auto-tag, release, unsafe reset, or bypass.

### 6. Hook gates block fake readiness

Delivery and verification gates now treat insufficient proof as blocking. They also keep degraded runtime status visible instead of allowing the report to call it PASS.

## What is still not claimed

This release does not claim formal test coverage proof.

It also does not claim Codex `exec` runtime hook execution is fully proven in every environment. If runtime is `DEGRADED`, SHK must report it as degraded and must not use it as release-ready runtime evidence.

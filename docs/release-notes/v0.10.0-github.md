# v0.10.0: Quality Gate Suite and Codex entry banner

This release makes SHK more than a set of working rules. It adds a machine-checkable quality gate: the agent can run checks, write structured evidence, and let hooks decide whether commit/tag-style actions are safe.

## What changed

### 1. One command can produce delivery evidence

```bash
node scripts/shk.js verify --risk release --write-evidence
```

`verify` writes both machine-readable and human-readable evidence:

- `.harness/verify-evidence.json`
- `.harness/verify-evidence.md`
- `docs/verification-report.md`

The important part is not the command itself. The important part is that SHK now has a durable answer to: “what exactly did we verify?”

### 2. Commit and tag gates read real evidence

`verification-gate.js` now prefers the JSON evidence from `shk verify`.

For normal delivery work, the gate expects `overall=READY`. For release tags, it requires stricter release-risk evidence. That reduces the chance of turning “I think it passed” into a permanent tag.

### 3. Security scan is built in

```bash
node scripts/shk.js security scan
```

The scanner looks for:

- common secret patterns;
- public-repo leak patterns from config;
- high-risk hook or MCP configuration.

The public kit does not ship private organization dictionaries. Project-specific leak rules should live in the target project, not in the public SHK repo.

### 4. Doctor can catch half-broken hook installs

`shk doctor` checks whether the harness wiring looks healthy, especially around:

- PreToolUse observations;
- Codex `UserPromptSubmit` entry wiring;
- banner script presence;
- recent entry banner records.

This makes “why did the harness not show up?” and “why did that tool call not get intercepted?” easier to debug.

### 5. Codex sessions now get a visible Harness banner

`harness-entry-banner.js` uses Codex’s `UserPromptSubmit` hook path to inject visible context into the model session.

The goal is simple: a new Codex session should know that Harness mode is active, instead of only having hooks run silently in the background.

### 6. Codex stage recovery is less fragile

Some Codex environments do not expose the same `Write` workflow as Claude Code. v0.10.0 adds a narrow recovery path for `.harness/current-stage.json` so a bad PLAN/stage state does not deadlock the session.

This is not a bypass. The path still validates stage name, time freshness, Infra Tier, and REVIEW gate rules.

### 7. Project hook sync no longer touches personal skills by accident

`update.sh` now supports:

```bash
--hooks-only <project>
--skip-skills
```

That means a project can refresh its hook scripts without rewriting the user’s global Claude/Codex skill directories.

## Who should upgrade

Upgrade if you want SHK to:

- produce a concrete READY / NOT_READY evidence file before delivery;
- make commit/tag gates depend on that evidence;
- scan for secrets and public-repo leak risks;
- debug hook installation problems faster;
- show the Harness banner in Codex sessions;
- sync project hooks without touching personal skills.

## Known limitation

Codex runtime smoke can still be `DEGRADED` in some environments. The smoke proves that Codex CLI can start a minimal run, but `codex exec` still does not fully prove that every project `.codex/hooks.json` command executed as intended.

Do not call this runtime PASS. Do not use it as release-ready runtime evidence.

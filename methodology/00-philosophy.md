# Harness Engineering 哲学

## 一句话

> 不要问"哪个模型更聪明"，要问"谁的执行系统更可靠"。

## 从回答系统到执行系统

过去两年，AI 对开发者的价值是**回答**：补全代码、回答问题、生成片段。人提问，模型回答，人决定怎么用。

2026 年正在发生的变化是**执行**：AI 拆解任务、调用工具、读写文件、运行测试、提交代码、持续推进目标。人给目标，Agent 拆解，Agent 执行，Agent 交付，人验收。

"回答系统"对组织的改变是增量的——每个人效率提升一点。
"执行系统"对组织的改变是结构性的——工作方式、协作模式、质量控制都要变。

## 三个关键概念

### Agent（智能体）

围绕目标展开行动的执行单元。不是聊天机器人，不是代码补全，是能自主完成"读代码→改代码→跑测试→提 PR"完整链路的执行者。

### Runtime（运行时）

Agent 运行、恢复、记录和受控协作的环境层。解决生命周期管理、状态持久化、工具调度、记忆、沙箱、错误恢复等问题。没有 Runtime，Agent 只是 Demo。

### Harness（执行壳）

把模型、工具、执行环境、记忆、审批流、子 Agent、消息系统连接起来的编排层。如果 Runtime 是操作系统，Harness 是让应用真正跑起来的那一层。

## Harness Engineering 的核心信念

### 1. Hook 强制 > LLM 自觉

LLM 的"遵守规则"本质上是概率性的。上下文越长，注意力越漂移，规则遵从度越低。

Hook 是工具级拦截，在 LLM 调用工具之前/之后强制执行检查。LLM 无法绕过、不会遗忘、不受上下文长度影响。

**规则写在 CLAUDE.md 里是"希望"，写在 Hook 里是"强制"。**

### 2. 独立 Agent > 长对话

长对话中 Agent 会逐渐"忘记"早期的约束和规则。每个任务启动独立 Agent：
- 上下文干净，rules 在 system prompt 最前面
- 没有历史对话噪声干扰
- Reviewer 和 Author 不是同一个 Agent，消除 author-bias

### 3. 多层自动化 QA > 单次检查

不依赖单一检查手段。5 层 QA 金字塔从自动到人工递进，AI 承担 4 层，人只做最后审查。每层都可以独立拦截问题，纵深防御。

### 4. 约束可追溯 > 口头约定

每条约束有唯一 ID（`C-{area}-{number}`），每次修复引用约束 ID，每次违规记录到 Violation History。规则不是空气中的共识，是有编号、有记录、可审计的工程制品。

### 5. 先规则后执行 > 先干后想

遇到问题，不是"赶紧修"，而是"先提炼规则，写入 constraints.md，再派 Agent 按规则修复"。规则沉淀下来，同类问题不再重复。

## Inner Loop / Outer Loop 模型

```
Outer Loop（人）
  设定目标 → 审查计划 → 最终验收 → 方向修正
       ↕
Inner Loop（AI）
  Plan → Setup → Execute → Verify → Review → Feedback
  （可以多次迭代，人不需要参与每次循环）
```

人管方向和最终质量，AI 管执行和过程质量。这不是"AI 替代人"，是"人机协作的正确分工"。

## 这个方法论不适合什么

- **一次性脚本/原型**：约束体系的 setup 成本大于收益
- **纯探索性研究**：方向不明确时不需要严格流程
- **没有测试框架的项目**：QA 金字塔依赖自动化测试
- **不用 AI 做开发的项目**：这是给"AI 参与开发"场景设计的

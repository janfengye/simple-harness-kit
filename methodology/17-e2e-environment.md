## 问题陈述

Experiment C (Planka) 实测发现：

> e2e 测试环境搭建占大量时间。init 时应检测项目的 docker-compose 和环境配置，生成测试环境快速启动指南。

具体痛点：
- AI 第一次面对一个 sample 项目，要花 30 分钟+ 摸清 "怎么把测试跑起来"
- 不同项目的 e2e 启动姿势天差地别 (npm test / docker compose up / make e2e / pytest --integration / ...)
- 缺少一份 "5 行命令把 e2e 跑起来" 的速查表
- AI 容易在 "依赖装不上" / "数据库连不上" / "端口冲突" 这类环境问题上消耗大量 context

M-13 要给 AI 一个**可重复的 e2e 环境搭建 playbook**，而不是每次都从头探索。

## 核心原则

1. **Init 时一次性侦测** — harness-init 必须在初始化阶段扫描 e2e 相关 marker 文件（docker-compose / Makefile / package.json scripts / .env.sample），生成 `.harness/e2e-quickstart.md` 速查表
2. **首次跑通即记录** — 不管是 AI 还是人，首次成功跑通 e2e 后必须把"实际工作的命令序列"记录到 `.harness/e2e-quickstart.md`，覆盖最初的 init 推断
3. **e2e 环境健康度是 Infra Tier 评估的输入** — 如果 e2e 跑不起来或要 30+ 分钟才能跑通，应该影响 Infra Tier 评分（H6 测试确定性的延伸）

## E2E Marker 检测清单

Init 时 harness-init skill 应该检测以下 marker 并据此生成 quickstart 指南：

| Marker | 含义 | 推断的 e2e 启动命令 |
|---|---|---|
| `docker-compose.yml` / `compose.yml` | Docker 编排 | `docker compose up -d && <wait_healthy> && <run_tests>` |
| `Dockerfile` | Docker 构建 | `docker build -t <name> . && docker run <name> npm test` |
| `Makefile` 含 `test` / `e2e` target | Make 工作流 | `make test` 或 `make e2e` |
| `package.json` scripts 含 `test:e2e` / `e2e` | npm script | `npm run test:e2e` |
| `pytest.ini` / `pyproject.toml [tool.pytest]` 含 `--integration` marker | Python pytest | `pytest -m integration` |
| `playwright.config.*` | Playwright UI e2e | `npx playwright test` |
| `cypress.config.*` | Cypress UI e2e | `npx cypress run` |
| `tests/e2e/` 或 `e2e/` 目录 | e2e 测试隔离 | 进入目录看具体框架 |
| `.env.sample` / `.env.example` | 需要 env vars | `cp .env.sample .env && <edit secrets>` 是先决步骤 |
| `db/migrations/` / `prisma/migrations/` | 数据库迁移 | e2e 前必须 `<migrate command>` |
| `docker-compose.test.yml` | 测试专用 compose | `docker compose -f docker-compose.test.yml up` |

## `.harness/e2e-quickstart.md` 生成模板

Init 时生成一份带 placeholder 的速查表：

```markdown
# E2E Quickstart (auto-generated by harness-init)

> 这份指南由 init 时扫描项目自动生成，基于 marker 文件的最优推断。
> **首次成功跑通 e2e 后, 请手动更新本文件**, 把推断的步骤替换成实际工作的命令序列。

## 检测到的 marker

- [x] docker-compose.yml
- [x] .env.sample
- [x] db/migrations/
- [ ] Makefile
- [ ] tests/e2e/

## 推断的启动序列

```bash
# Step 1: 准备 env
cp .env.sample .env
# 编辑 .env 填入 secrets (DATABASE_URL / API_KEY / ...)

# Step 2: 启动依赖
docker compose up -d

# Step 3: 等服务 healthy
sleep 10  # 或者更精确: docker compose ps | grep healthy
# (项目作者: 替换成实际 healthcheck)

# Step 4: 跑迁移
npm run db:migrate  # 或 yarn db:migrate / prisma migrate deploy

# Step 5: 跑 e2e 测试
npm run test:e2e  # 或具体 framework 命令

# Step 6: 收尾
docker compose down -v  # 清理临时数据
```

```

## 实战 case study (Planka)

Planka 的 e2e marker 检测：

| Marker | 在 Planka 里 |
|---|:---:|
| `docker-compose.yml` | ✅ |
| `Dockerfile` | ✅ |
| `Makefile` | ❌ |
| `package.json` scripts.test | ✅ (`npm run server:test && npm run client:test`) |
| `package.json` scripts.test:e2e | ❌ |
| `pytest.ini` | ❌ |
| `playwright.config.*` | ❌ (Experiment C 用了 Playwright 但单独跑) |
| `tests/e2e/` 目录 | ❌ |
| `.env.sample` | ✅ (`server/.env.sample`) |
| `db/migrations/` | ✅ (`server/db/migrations/`) |

推断的 quickstart:

```bash
# Planka e2e quickstart (推断版)
cd server
cp .env.sample .env
# 编辑 .env 填 BASE_URL / DATABASE_URL / SECRET_KEY (见 .env.sample)
docker compose up -d postgres  # Planka 用 PostgreSQL
npm install
npm run db:migrate
cd ..
npm test  # 跑 server 和 client tests
```

但 Experiment C 实战验证发现以下额外坑（首次跑通后才知道）：

- `lifecycle.test.js` `this.timeout(5000)` 太短 → 改为 60000 (#11 已修)
- `utils/remote-address.js` 耦合 sails 全局 `_` → 显式 require lodash (#11 已修)
- `User.test.js` 整个被注释 → 重写为 12 个 unit tests (#11 已修)

这些坑应该被 sequence "首次跑通后填到 quickstart 已知坑段"。Planka case study 是 [examples/experiment-c-planka/m12-baseline.md](../examples/experiment-c-planka/m12-baseline.md) 的实际产物。

## 与 init-prompt 的衔接

`init-prompt.md` 应该在 init step 中加入：

> **Step N: e2e 环境侦测 (M-13)**
> - 扫上面的 marker 清单
> - 生成 `.harness/e2e-quickstart.md` 推断版
> - 不要尝试真的跑 e2e (会消耗大量 context, 留给后续 session)
> - 提醒用户 "首次跑通后请更新 quickstart"

## 与 Infra Tier 的衔接

M-13 的 e2e quickstart 是 [16-infra-tier.md](./16-infra-tier.md) 的辅助工具，不替代 Infra Tier 评估。但两者协同：

- e2e 跑得起来 = H1 + H2 部分通过的证据
- e2e 速度 < N 分钟 = H6 (确定性) 的间接证据
- e2e quickstart 完整 = 新贡献者第一天就能 contribute 的前提

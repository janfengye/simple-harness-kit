# Session Log：全过程记录机制

## 为什么要记录

没有完整记录，就无法：
1. 验证 Harness 是否真的在发挥作用
2. 找到方法论与现实的偏差
3. 对 Harness Kit 本身做 Feedback Loop 优化
4. 给后续使用者提供可信的操作参考

## 记录什么

| 事件类型 | 记录内容 | 价值 |
|---------|---------|------|
| **人的指示** | 原话，不解读不简化 | 理解真实意图和表达方式 |
| **AI 的决策** | 做了什么 + 为什么 + 依据哪篇方法论 | 验证方法论是否被正确应用 |
| **AI 的产出** | 生成/修改了什么文件，关键内容摘要 | 可追溯的交付链 |
| **Hook 事件** | 哪个 Hook 触发、原因、结果（阻止/警告/放行） | 验证 Hook 有效性 |
| **Gate 判定** | 通过/未通过 + 具体数据 | 验证 Gate 条件是否合理 |
| **人的反馈** | 原话 + F1-F5 处理过程 | 方法论改进的直接输入 |
| **偏差记录** | 方法论说应该 X，实际做了 Y，为什么 | **最有价值**——方法论与现实的差距 |

## 日志格式

文件位置：`.harness/session-log.md`

```markdown
# Harness Session Log

## 元信息
- 项目: {项目名}
- 需求: {一句话描述}
- 开始时间: {YYYY-MM-DD HH:MM}
- Harness Kit 版本: {commit hash 或 tag}
- AI 工具: {Claude Code / Codex / 其他}
- 模型: {具体模型}

---

## 事件记录

### [{HH:MM}] 阶段: {PLAN/SETUP/EXECUTE/VERIFY/REVIEW/FEEDBACK}
**类型:** {人的指示 / AI 决策 / AI 产出 / Hook 事件 / Gate 判定 / 人的反馈 / 偏差}

{内容}

---
```

## 各事件类型的记录模板

### 人的指示

```markdown
### [{HH:MM}] 阶段: {阶段} | 人的指示
> {原话，一字不改}
```

### AI 决策

```markdown
### [{HH:MM}] 阶段: {阶段} | AI 决策
- **动作**: {做了什么}
- **原因**: {为什么这么做}
- **依据**: {引用 methodology/ 中的具体章节}
```

### AI 产出

```markdown
### [{HH:MM}] 阶段: {阶段} | AI 产出
- **文件变更**:
  - 创建: {文件列表}
  - 修改: {文件列表}
  - 删除: {文件列表}
- **内容摘要**: {关键产出的一句话描述}
```

### Hook 事件

```markdown
### [{HH:MM}] 阶段: {阶段} | Hook 事件
- **Hook**: {hook 名称}
- **触发命令/操作**: {触发 Hook 的具体操作}
- **结果**: {阻止(exit 2) / 警告(stderr) / 放行}
- **AI 响应**: {AI 如何处理这个 Hook 信号}
```

### Gate 判定

```markdown
### [{HH:MM}] 阶段: {阶段} | Gate 判定
- **Gate**: {哪个阶段的 Gate}
- **结果**: PASS / FAIL
- **详情**:
  | 检查项 | 结果 | 数据 |
  |--------|------|------|
  | Build | PASS | exit 0 |
  | Test | PASS | 5/5 passed, 87% coverage |
  | ... | ... | ... |
```

### 人的反馈

```markdown
### [{HH:MM}] 阶段: FEEDBACK | 人的反馈
> {原话}

**F1 记录**: {原话}
**F2 分类**: {规则层/工具层/配置层/实例层}
**F3 规则**: {提炼出的规则}
**F4 写入**: {Constraint ID}
**F5 派发**: {Agent 修复动作}
```

### 偏差记录

```markdown
### [{HH:MM}] 阶段: {阶段} | 偏差
- **方法论要求**: {methodology/ 中说应该怎么做}
- **实际执行**: {实际做了什么}
- **偏差原因**: {为什么偏离——方法论不合理 / 场景不适用 / Agent 遗忘 / 其他}
- **建议**: {对方法论的修改建议}
```

## AI 的记录职责

执行 Harness 的 AI 必须在每个关键动作后自动追加日志。具体规则：

1. **每次收到人的指示** → 记录原话
2. **每次做出决策** → 记录动作 + 原因 + 依据
3. **每次 Hook 触发** → 记录 Hook 事件
4. **每次 Gate 判定** → 记录完整结果
5. **每次收到反馈** → 记录 F1-F5 全过程
6. **每次偏离方法论** → 记录偏差

**关键约束：先记录，再行动。** 不是做完回头补，是做之前/做的时候就记。

## 日志的用途

### 用途 1：实验证据

实验完成后，session-log.md 直接复制到 `examples/` 作为证据，不需要事后重新整理。

### 用途 2：方法论优化

提取所有偏差记录，分析模式：
- 同一类偏差反复出现 → 方法论需要修改
- 某个 Hook 频繁误触发 → 模板需要改进
- 某个 Gate 总是被跳过 → Gate 条件可能不合理
- 某个阶段总被省略 → 可能真的不需要

### 用途 3：操作指导

新用户读 session-log，能看到：
- 真实的操作过程（不是理想化的流程图）
- 遇到了什么问题、怎么解决的
- 哪些 Hook 真的拦住了问题
- 人是怎么与 AI 交互的

## 在 Harness Init 时自动创建

`harness-init` skill 应在初始化时创建 `.harness/session-log.md`，写入元信息头。

在 `.claude/rules/` 中加一条规则提醒 AI 记录：

```markdown
# Session Log 规则

每次执行关键动作时，自动追加记录到 .harness/session-log.md。
记录类型包括：人的指示、AI 决策、AI 产出、Hook 事件、Gate 判定、人的反馈、偏差。
先记录，再行动。
```

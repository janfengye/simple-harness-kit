# 反模式与 Red Flags

## 流程反模式

### 1. "先干后想"

```
❌ 发现问题 → 直接修 → 下次同样问题再修一次
✅ 发现问题 → F1-F5 → 提炼规则 → 派 Agent 按规则修 → 下次不再出现
```

### 2. "一个 Agent 全搞定"

```
❌ 一个长对话里完成：需求分析 + 实现 + 测试 + review + 部署
   → 后期规则遗忘，author-bias，上下文污染

✅ 每个任务独立 Agent
   → 上下文干净，规则有效，reviewer 独立于 author
```

### 3. "QA 就是跑个测试"

```
❌ 测试通过 = 质量合格
   → 测试只覆盖 Layer 2，遗漏 spec 合规性和设计问题

✅ 5 层 QA 金字塔
   → 每层检查不同维度，纵深防御
```

### 4. "Hook 太烦了"

```
❌ 关掉 Hook → "我会记得遵守规则的"
   → 长对话后 100% 会忘

✅ Hook 是保护不是限制
   → 就像安全带，不舒服但救命
```

### 5. "这次特殊"

```
❌ "这个很简单，不需要走流程"
   → 简单的事情最容易出错，因为缺少审慎

✅ 简化模式（轻量/标准/完整）
   → 按复杂度选择流程深度，但不跳过
```

### 6. "主观 QA"

```
❌ "看起来不错" → 交付
   → 无法复现判断标准，不同人标准不同

✅ 量化指标 → 交付
   → 覆盖率 80%、构建成功、lint 0 error、spec 逐项 PASS
```

## Agent 使用反模式

### 7. "在 prompt 里写实现代码"

```
❌ Director 在 Agent prompt 中：
   "把第 42 行改成 const x = foo.bar()"

✅ Director 在 Agent prompt 中：
   "参考 C-API-02，修复 API 响应格式不一致的问题"
```

Director 定规则，Agent 定实现。

### 8. "让 Agent 读文件而不是给文本"

```
❌ "Read docs/plan.md and implement Task 3"
   → 多一轮读取，可能读错，上下文不确定

✅ 直接把 Task 3 的完整文本嵌入 prompt
   → 上下文明确，Agent 立即开始
```

### 9. "忽略 Agent 的 BLOCKED 状态"

```
❌ Agent 返回 BLOCKED → 重新派同一个 Agent 重试
   → 同样的问题同样的结果

✅ Agent 返回 BLOCKED → 分析原因 → 上下文不够？任务太大？计划有误？
   → 针对性解决后再派发
```

### 10. "修复类 Agent 不引用 Constraint ID"

```
❌ "修复登录页面的按钮问题"
   → 修完这一个，下一个同类问题还会出现

✅ "修复 C-UI-03 违规，读取 constraints.md 中的完整描述"
   → 按规则修所有同类问题
```

## Red Flags 清单

出现以下思维/言语模式时，说明正在合理化违规。**立即停下来，回到流程。**

### 跳过流程的信号

- "这个很简单，不需要..."
- "先提交，后面再..."
- "时间紧，先过了再说"
- "手动验证过了"（没有命令输出证据）
- "测试应该能通过"（没有运行）
- "只是小改动"

### 跳过 QA 的信号

- "看起来不错"
- "应该没问题"
- "这部分不需要测试"
- "之前跑过了"（不是本次变更后跑的）

### Agent 滥用的信号

- "我来直接改一下"（Director 越权写代码）
- "这太简单了不需要派 Agent"
- "让它再试一次"（没有分析失败原因）

### 上下文劣化的信号

- Agent 开始忘记早期的约束
- Agent 输出质量明显下降
- Agent 开始重复之前已经做过的事
- Agent 说"这种情况应该例外"

→ 需要 compact 或 启动新 Agent

## 处理 Red Flag

```
发现 Red Flag
    ↓
停下当前操作
    ↓
识别正在跳过哪个阶段/Gate
    ↓
回到该阶段，按流程执行
    ↓
记录到 Violation History（如果是新类型的违规）
```
